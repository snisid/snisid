# SNISID: Event Ordering, Consistency & Deduplication

For a national identity platform, data consistency and exact ordering are non-negotiable. This architecture defines the mathematical and operational controls that guarantee every identity event is processed exactly once and in the correct sequence.

---

## 1. Strong Ordering Architecture

SNISID ensures that events relating to a specific citizen are always processed in their temporal sequence.

### 1.1. Partition-Key Strategy
- **Key Determinism**: All events related to a specific `identity_id` (NID) MUST use that ID as the Kafka Partition Key. 
- **Effect**: Kafka guarantees that all events for that key are routed to the **Exact Same Partition**, ensuring strict sequential delivery to consumers.

### 1.2. Producer-Side Sequencing
- **Sequence IDs**: Every event includes an incrementing `sequence_number` generated by the originating microservice (e.g., `identity-api`).
- **Gap Detection**: Consumers track the last processed `sequence_number` for a given `identity_id`. If a gap is detected (e.g., received sequence 10 after 8), the consumer triggers a "Consistency Alert" and pauses consumption for that partition to prevent out-of-order state corruption.

---

## 2. Distributed Consistency Model

SNISID implements **Exactly-Once Semantics (EOS)** across regional clusters.

- **Transactional Producers**: Producers use `enable.idempotence=true` and `transactional.id` to ensure that a batch of messages is either all committed or none.
- **Cross-Region Consistency**: 
  - **Synchronous Replication**: For critical state changes (e.g., `Identity Revocation`), producers wait for ack from both the Primary and DR clusters.
  - **Determinism**: Every event includes a **Sovereign Timestamp** (PTP-synchronized), which is used as the tie-breaker in "Last Write Wins" scenarios during regional failover.

---

## 3. Real-Time Event Deduplication

To prevent fraud and technical errors (e.g., double-clicks or network retries), SNISID employs a **Multi-Layered Deduplication Strategy**.

### 3.1. Idempotent Consumer Design
Every state-changing microservice is designed as an **Idempotent Sink**:
- **Operation Hashing**: Before processing, the service calculates a **SHA-256 Fingerprint** of the event payload + `correlation_id`.
- **Deduplication Cache**: The fingerprint is checked against a high-speed **Redis Bloom Filter** (30-day window).
- **Database Enforcement**: A unique constraint on the `event_fingerprint` column in the destination database (PostgreSQL/Neo4j) acts as the final safety net.

### 3.2. Deduplication Window
- **Sliding Window**: For high-volume streaming (e.g., Auth logs), Flink uses a 5-minute sliding window to deduplicate identical events from a single source IP or Device ID.

---

## 4. Replay Determinism & Recovery

- **The Idempotency Key**: When replaying billions of historical events (the "Time Machine"), the `event_id` and `fingerprint` ensure that the system does not trigger duplicate alerts or fraud cases for historical data already processed.
- **Side-Effect Management**: During a replay, "Side-Effect" actions (e.g., sending an SMS or triggering a real-world alert) are **Muted** by the processing engine, while the internal state (Graph/DB) is updated perfectly.

---

## 5. Failure Recovery Workflows

| Failure Scenario | Resolution Mechanism |
| :--- | :--- |
| **Out-of-Order Delivery** | Partition-key routing + Sequence ID validation at the consumer layer. |
| **Duplicate Event Ingestion** | Redis Bloom Filter + Unique DB Constraints + Kafka Idempotence. |
| **Regional Data Lag** | "Sovereign Timestamp" tie-breaking + MirrorMaker offset translation. |
| **Partition Rebalance** | Sticky Assignors to minimize state migration and maintain ordering during consumer scaling. |
