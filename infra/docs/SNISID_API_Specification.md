# SNISID: API Specification & Standards

This document establishes the official API Contracts for the SNISID platform. It defines how external agencies interact with the API Gateway (REST) and how internal microservices interact with each other synchronously (gRPC).

## 1. REST API Standards (External)

All external-facing APIs exposed through the Gateway must strictly adhere to the following RESTful standards.

### 1.1. Base URL and Versioning
*   **Format:** `https://api.snisid.gov/{domain}/v{major_version}/{resource}`
*   **Example:** `https://api.snisid.gov/identity/v1/citizens`

### 1.2. Authentication & Authorization Headers
All requests must include standard security headers verified by the API Gateway.
*   `Authorization: Bearer <JWT>`
*   `X-Agency-ID: <UUID>`
*   `X-Correlation-ID: <UUID>` (Optional, generated by Gateway if missing).

### 1.3. Standard Response Envelope
To ensure predictability for consuming agencies, all successful responses must wrap data in a standard envelope.

```json
{
  "status": "success",
  "data": { ... },
  "meta": {
    "timestamp": "2026-05-07T14:30:00Z",
    "correlation_id": "req-1234-abcd"
  }
}
```

### 1.4. Pagination Rules
Endpoints returning collections (e.g., searching for fraud cases) must use cursor-based pagination for high performance on large datasets.

**Request:** `GET /fraud/v1/cases?limit=50&cursor=eyJpZ...`
**Response Meta Block:**
```json
"meta": {
  "pagination": {
    "next_cursor": "eyJpZ...",
    "has_more": true
  }
}
```

### 1.5. Standardized Error Models (RFC 7807)
Errors must follow the `Problem Details for HTTP APIs` standard.

```json
{
  "type": "https://api.snisid.gov/errors/validation-failed",
  "title": "Validation Failed",
  "status": 400,
  "detail": "The provided national_id format is invalid.",
  "instance": "/identity/v1/citizens",
  "correlation_id": "req-1234-abcd",
  "invalid_params": [
    {
      "name": "national_id",
      "reason": "Must be exactly 14 numeric characters."
    }
  ]
}
```

## 2. Core API Endpoints (OpenAPI Highlights)

### Identity Domain
*   `POST /identity/v1/citizens` - Enrolls a new citizen (Async, returns `202 Accepted` with a Job ID).
*   `GET /identity/v1/citizens/{id}` - Retrieves full demographic data (Requires high clearance).
*   `PATCH /identity/v1/citizens/{id}/status` - Updates lifecycle status (e.g., suspending an ID).

### Fraud & Intelligence Domain
*   `POST /fraud/v1/assessments` - Submits a transaction for synchronous risk scoring.
*   `GET /intelligence/v1/graph/clusters/{id}` - Returns graph edges for a specific identity cluster.

## 3. Internal gRPC Services (Protobuf)

Inside the secure perimeter (Zone 0), microservices communicate synchronously via high-performance gRPC over mTLS.

### Example: Fraud Scoring Service calling AI Inference Service
```protobuf
syntax = "proto3";
package snisid.ai.v1;

service BiometricInference {
  // Synchronous 1:1 match for immediate verification
  rpc VerifyFace(VerifyRequest) returns (VerifyResponse);
}

message VerifyRequest {
  string correlation_id = 1;
  bytes probe_image_data = 2; // Raw encrypted bytes
  string reference_biometric_profile_id = 3;
}

message VerifyResponse {
  float confidence_score = 1; // 0.0 to 1.0
  float liveness_score = 2;
  float deepfake_score = 3;
  bool match_confirmed = 4; // Based on strict national threshold
}
```

## 4. Backward Compatibility Rules

1.  **Never rename or remove a field** in a published API.
2.  **Never change a field's type** (e.g., String to Integer).
3.  **New fields must be optional.** If a new feature requires mandatory data, a new Major Version (`v2`) must be created.
4.  Clients must be designed to silently ignore unknown fields in the JSON response.
