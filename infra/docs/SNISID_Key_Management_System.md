# SNISID: Production-Grade Key Management System (KMS)

The KMS is the cryptographic heart of the SNISID platform. It provides the Hardware Root of Trust for all identity, communication, and data protection services, ensuring that the "Keys to the Nation" are governed with absolute integrity and resilience.

---

## 1. KMS Architecture: The Sovereign Vault Core

SNISID utilizes **HashiCorp Vault** in an Enterprise configuration, hardened with physical **Hardware Security Modules (HSMs)**.

### 1.1. Cluster Design
- **Consensus**: Vault uses **Raft Integrated Storage** for sub-10ms key retrieval and high availability.
- **Topology**: 5-node primary cluster with **Performance Replication** across 3 national data centers (Zones).
- **Physical Isolation**: The KMS cluster is hosted in a dedicated, air-gapped management network, accessible only via the **National API Gateway**.

### 1.2. HSM Integration (Root of Trust)
- **Standard**: FIPS 140-2 Level 3 compliant HSMs.
- **Auto-Unseal**: Vault is configured to automatically unseal using the HSM. The HSM holds the master wrapping key, ensuring that the KMS cannot boot without the physical presence of the hardware module.

## 2. Key Lifecycle Management

| Phase | Technical Enforcement |
| :--- | :--- |
| **Generation** | Keys are generated using a hardware-based Random Number Generator (RNG) inside the HSM. |
| **Rotation** | Automated 90-day rotation for Master Keys (KEKs). Data Encryption Keys (DEKs) are rotated per-record. |
| **Revocation** | Instant invalidation of any key via the KMS API. Triggers immediate re-encryption or crypto-shredding. |

| **Escrow** | Secure, offline storage of key fragments for legal/judicial recovery, protected by multi-party approval. |
| **Destruction** | Permanent purging of keys from HSM memory and disk, following DoD-standard wiping procedures. |

## 3. Trust Hierarchy & Envelope Encryption

To scale encryption to millions of identities, SNISID uses **Envelope Encryption**.

1. **Root Master Key (RMK)**: Stays inside the HSM.
2. **Key Encryption Key (KEK)**: Generated by Vault per-agency (tenant). Encrypted by the RMK.
3. **Data Encryption Key (DEK)**: Generated per-identity or per-transaction. Encrypted by the KEK.
4. **Ciphertext**: The actual citizen data, encrypted by the DEK.

**Benefit**: If a single DEK is stolen, only one record is at risk. If a KEK is rotated, only the DEKs for that agency need to be re-wrapped, not the entire national database.

## 4. Distributed Trust (Shamir’s Secret Sharing)

For catastrophic scenarios (e.g., HSM failure or total data center loss), the KMS employs a **Shamir’s Secret Sharing** scheme for the Master Unseal Key.

- **Threshold**: 3 of 5 (3/5).
- **Key Holders**:
  1. Minister of National Security
  2. Chief of Police
  3. Director of Civil Registry
  4. Chief Technology Officer (CTO)
  5. Independent National Auditor
- **Protocol**: Unsealing the core requires the physical presence and hardware keys of any three of these individuals.

---

## 5. Multi-Region Resilience & Recovery

- **Disaster Recovery (DR) Replication**: A warm-standby cluster in a secondary region continuously synchronizes state.
- **Failover**: Automatic redirection of Identity and Data services to the secondary KMS cluster in < 30 seconds during a primary site failure.
- **Recovery Time Objective (RTO)**: 5 Minutes.
- **Recovery Point Objective (RPO)**: 0 (Synchronous replication).

---

## 6. Security Controls & Audit

- **Identity-Bound Key Access**: Keys are never granted to users. They are granted to **SPIFFE IDs** (Workloads). A service must prove its identity via SPIRE before Vault will release a DEK or decrypt a payload.
- **Transit Engine (Encryption-as-a-Service)**: High-sensitivity microservices use the "Transit Engine" to encrypt data without ever touching the raw keys.
- **Immutable Audit**: Every key creation, rotation, and usage is logged to the **Sovereign Audit Ledger**, including the `correlation_id` of the requesting microservice.
