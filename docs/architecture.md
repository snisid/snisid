# SNISID Architecture

## Overview
SNISID is a modular system composed of four primary backend services, a REST API gateway, and a web frontend. The system is designed for both online and offline operation, with a sync agent bridging disconnected field data collection with a central server.

## Core Services
The core services layer (`snisid-svc.exe`) manages data ingestion, processing pipelines, and background job scheduling. It consists of a Windows/Linux service that orchestrates internal modules including data validation, transformation, and storage. This layer is the single source of truth for all persistent state and communicates with other components via named pipes or local TCP.

## API Gateway
The REST API (`snisid-api`) exposes all CRUD operations, administrative functions, and health/status endpoints over HTTPS. It is built as a standalone HTTP server (Go/ASP.NET) that authenticates requests via JWT or client certificates. The API layer translates between external HTTP clients and internal core service protocols.

## Frontend
The frontend is a single-page application (HTML/JS/CSS or React/Vue) served by IIS (Windows) or nginx (Linux). It communicates exclusively with the API Gateway and provides dashboards, data entry forms, and administrative panels. No business logic resides in the frontend — it is a pure presentation layer.

## Sync Agent
The `sync-agent.py` offline-to-online synchronization bridge watches a local directory for data files generated by disconnected field tools. It hashes each file, runs it through a configurable ETL pipeline, archives the processed file, and records sync state as JSON. Conflicts are resolved via last-write-wins by default, with all resolutions logged for audit.

## Offline / WinPE Mode
For disaster recovery or field operations without network connectivity, SNISID can be booted from a WinPE ISO containing a self-contained offline viewer and data entry tools. Data collected offline is stored in portable SQLite databases; when connectivity is restored, the sync agent ingests these files into the main system.

## Deployment
SNISID supports three deployment targets: Windows Server (IIS + Windows Services), Linux (nginx + systemd), and air-gapped/offline via WinPE. The MSI and NSIS installer packages handle Windows deployment; a deploy.ps1 script automates server provisioning. Linux deployments use distro-appropriate packaging (deb/rpm) managed by Ansible or Salt.
